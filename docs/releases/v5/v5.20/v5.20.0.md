## Description of Changes

<!-- user-editable-section-start -->
- Add some type-narrowing helper functions.
  - `isNonNullableObject` - checks if the input is a non-null object and narrows the type accordingly.
  - `objectContainsKeys` - checks whether an object contains all specified keys and narrows the type to include those keys.
  - `containsKeys` - combines both checks, allowing an unknown value to be safely narrowed to an object containing the specified keys.
- In all of the above cases, the input type will be narrowed down after the check, ensuring type safety when working with TypeScript.

## Additional Notes

- These functions are particularly useful in our `check` methods on our error classes. For example, our `DataError` check method before used to look like this:

```typescript
class DataError extends CodeError {
  public static override check<
    DataType extends object = Record<PropertyKey, unknown>,
    ErrorCode extends string = string,
  >(input: unknown): input is DataError<DataType, ErrorCode> {
    if (input instanceof DataError) {
      return true;
    }

    return (
      typeof input === "object" &&
      input !== null &&
      "message" in input &&
      typeof input.message === "string" &&
      "code" in input &&
      typeof input.code === "string" &&
      "data" in input &&
      typeof input.data === "object" &&
      input.data !== null
    );
  }
}
```

And now it looks like this:

```typescript
class DataError extends CodeError {
  public static override check<
    DataType extends object = Record<PropertyKey, unknown>,
    ErrorCode extends string = string,
  >(input: unknown): input is DataError<DataType, ErrorCode> {
    if (input instanceof DataError) {
      return true;
    }

    return (
      containsKeys(input, ["data", "code", "message"]) &&
      typeof input.message === "string" &&
      typeof input.code === "string" &&
      isNonNullableObject(input.data)
    );
  }
}
```

- It's especially helpful as it makes proper type narrowing a lot easier to do, and the easier it is to do proper type-narrowing, the less likely people will be to reach for the `any` type, and therefore the more type-safe the code becomes.
